\section{Komponenten des Verfahrens}\label{chap:Komponenten}
Das vorgestellte Verfahren besteht aus den folgenden Komponenten:
\begin{itemize}
    \item \texttt{testDiscovery}: Liefert die Gesamtheit aller Testcases
    \item \texttt{test archive}: Enthält die Ausgabe aller Testdurchläufe, die für den aktuellen Stand beachtet werden sollen
    \item \texttt{testAuditor}: Evaluiert, welche Testcases noch ausgeführt werden müssen
    \item \texttt{testExecution}: Führt eine gegebene Menge von Testcases aus
    \item \texttt{controller}: Steuert die Interaktion mit dem user, sowie die Kommunikation von Komponenten untereinander
    \item \texttt{user}: Interagiert nach eigenem Ermessen mit dem controller
\end{itemize}

\subsection{\texttt{testDiscovery}}\label{chap:Komponente_testDiscovery}
Durch die Wahl für JUnit XML als Ausgabeformat, das vom Verfahren verstanden werden soll, müssen die in \chapref{Anforderungen Testausgabeformat} aufgelisteten Anforderungen, die von JUnit XML nicht erfüllt werden, anderweitig umgesetzt werden, damit alle in \chapref{Anforderungen Tool} gesetzten Ziele erreicht werden können.\\
Die größte Herausforderung ist, dass aus einer JUnit-XML Ausgabe nur jene Testcases ermittelt werden können, welche während des Testlaufes ausgeführt wurden. \texttt{testAuditor} benötigt jedoch zwingend Informationen über die Gesamtheit aller Testcases im Projekt.

Diesem Zweck dient die \texttt{testDiscovery:}\\
Bei dieser Komponente ist die Implementierung als shellscript am geeignetsten, da sie bei einem Aufruf die Gesamtheit aller Testcases des Projektes in einem definierten Format zurückgeben soll. Manche Testframeworks bieten Zugriff auf ihre interne Discovery für Testcases, was in diesen Fällen die Implementation recht einfach gestaltet. 

\subsubsection{Format der Ausgabe der \texttt{testDiscovery}}\label{chap:Komponente_testDiscovery_format}
\texttt{testAuditor} erwartet die Ausgabe der \texttt{testDiscovery} in diesem Format:
\begin{lstlisting}[caption={Format der \texttt{testDiscovery}}, label={TAPmin}]
<?xml version="1.0" encoding="utf-8"?>
<testsuite>
    <testcase classname="" name="" qualifiedName=""/>
    <testcase classname="" name="" qualifiedName=""/>
</testsuite>
\end{lstlisting}
Der Inhalt der beiden tags ''classname'' und ''name'' für einen Testcase muss dabei zwingend identisch mit dem Inhalt der beiden tags sein, wie er in der JUnit-XML Ausgabe des Testcase steht, da in einem späterem Schritt über die Kombination dieser beiden tags die Testcases eindeutig im JUnit XML identifiziert werden.\\
\texttt{qualifiedName} enthält eine Zeichenfolge, über die das Testframework einen Testcase identifizieren, und ausführen kann. Je nach Testframework werden dazu der Name des Testcase, Dateipfade, Namen von Klassen oder Testsuites und/oder weitere Informationen benötigt. \texttt{qualifiedName} dient in dem Schritt der \texttt{testExecution} (siehe \chapref{Komponente_testExecution}) bei der Ausführung einzelner Testcases.

Die \texttt{testDiscovery} ist hochgradig vom verwendeten Testframework abhängig, da die spezielle Implementierung der Testausführung des jeweiligen Testframeworks beachtet werden muss. Daher muss diese Komponente für jedes Softwareprojekt eigens angepasst werden.

\subsection{\texttt{test archive}}\label{chap:Komponente_archive}
Das \texttt{test archive}, kurz \texttt{archive}, beschreibt die Komponente des Verfahrens, welche die in JUnit XML festgehaltenen Testausgaben von vorherigen Testläufen beeinhaltet, die \texttt{testAuditor} als Input zur Verfügung stehen.\\
Ein Projekt muss so konfiguriert werden, dass sichergestellt ist, dass keine Dateien aus dem \texttt{archive}, die sich auf einen alten Stand des Projekts beziehen, vom Verfahren genutzt werden. Hierfür bietet sich die Implementierung des \texttt{archive} über git notes an, wie im folgenden Abschnitt untersucht wird.

\subsubsection{git notes}
Eine wenig bekannte und selten genutzte Funktionalität von Git ist, dass an einen Commit sogenannte git notes angehängt werden können. Der Inhalt dieser notes wird separat vom eigentlichen Inhalt des Commits verwaltet, der Hash eines Commits wird nicht durch Veränderungen der notes beeinflusst. Dadurch können einem Commit zusätzliche Informationen hinzugefügt werden, ohne die Historie des Repositories zu verändern.\\
Dadurch bieten sich git notes in zweierlei Hinsicht für die Rolle des \texttt{test archive} an:\\
Zum einen wird das eigentliche Projekt nicht durch zusätzliche Dateien vom Verfahren verändert. Zum anderen gehört jede note zu genau einem Commit, wodurch vermieden werden kann, dass das Verfahren Dateien aus dem \texttt{archive} für einen vorherigen Commit erhält, da die notes für jeden Commit neu erstellt werden müssen.

\subsection{\texttt{testAuditor}}\label{Komponente_testAuditor}
Die Funktionalität von \texttt{testAuditor} als Kern des Verfahrens lässt sich am Besten mit einer einfachen Mengengleichung beschreiben:\\
Sei $T_{ges}$ die Menge aller Testcases eines Projektes, $T_{run}$ die Menge aller Testcases, für die ein Ergebnis vorliegt. \texttt{testAuditor} berechnet $T_{ges}-T_{run}=T_{todo}$, wobei $T_{todo}$ die Menge aller Testcases ist, für die noch kein Ergebnis vorliegt. Da nach \chapref{chap:limit_scope} von der Korrektheit und Vollständigkeit von $T_{run}$ ausgegangen werden darf, ist trivial, dass $T_{run}\subset T_{ges}$, sowie $T_{todo}\subset T_{ges}$.\\
Da $T_{run}+T_{todo}=T_{ges}$, ist die komplette Abdeckung aller Testcases garantiert, sofern eine fehlerlose Impementierung der Komponenten vorliegt.\\
$T_{ges}$ wird von \texttt{testDiscovery} bereitgestellt, während das \texttt{test archive} $T_{run}$ enthält. Beide Datenmengen werden \texttt{testAuditor} vom \texttt{controller} über ein in \chapref{chap:format_testAuditor} definiertes Format übermittelt.\\
Zu guter Letzt gibt \texttt{testAuditor} eine Menge von \texttt{qualifiedNames} an den \texttt{controller} zurück. Diese Menge beschreibt $T_{todo}$. 

\subsubsection{Format zur Kommunikation mit \texttt{testAuditor}}\label{chap:format_testAuditor}
Das Verfahren nutzt einen einfachen Wrapper um die Ausgabe von \texttt{testDiscovery} und die Dateien aus \texttt{archive}, um den Input für \texttt{testAuditor} bereitzustellen. 
\begin{lstlisting}[caption={Kommunikationsformat für \texttt{testAuditor}}, label={testAuditor_Format}]
<?xml version="1.0" encoding="utf-8"?>
<testAuditorInput version="1.0">
    <testDiscovery>
        <![CDATA[...]]>
    </testDiscovery>
    <reports>
        <report format="junit-xml">
            <![CDATA[...]]>
        </report>
    </reports>
</testAuditorInput>
\end{lstlisting}
Dieses Format muss vom \texttt{controller} bereitgestellt werden. \texttt{testAuditor} erwartet den Input auf \texttt{stdout}. 

\subsubsection{Parser für JUnit XML}
Das Parsen des vorangehend beschriebenen Kommunikationsformat, sowie des in \chapref{chap:Komponente_testDiscovery_format} beschriebenen Format der \texttt{testDiscovery} ist trivial.\\
Interessanter ist die Umsetzung eines Parsers für JUnit XML für die Daten aus dem \texttt{test archive}. Wie in \chapref{JUNIT} behandelt, besitzt JUnit XML kein standartisiertes Schema. Dementsprechend muss ein Parser für solche Dateien auf eine Art und Weise konfiguriert werden, die ohne eine feste erwartbare Struktur auskommt. Dafür wird bei der Implementation des Parsers ein streaming-basierter Ansatz verwendet:\\
Die XML-Datei wird sequenziell Element für Element eingelesen. Sobald ein Element mit dem namen \texttt{<testcase>} gefunden wird, wird überprüft, ob dieses Element die Attribute \texttt{name} und \texttt{classname} enthält. Ist dies der Fall, so wird das Element zu der Menge an bereits durchgeführten Testcases hinzugefügt.\\
Durch diese Art der Implementierung können zwar auch Testausgaben in JUnit XML geparst werden, welche die Attribute \texttt{name} und/oder \texttt{classname} nicht benutzen (siehe \lstref{JUNITcodeception}), jedoch wird durch das Fehlen von Attributen jeder Testcase als nicht ausgeführt betrachtet, was die Anwendung des Verfahrens hinfällig macht, da kein Testcase als durchgeführt betrachtet wird und in jedem Durchlauf des Verfahrens jeder Testcase ausgeführt wird. 

Für die genau Implementierung des Parsers im Code, siehe das in \chapref{Links} verlinkte Repository für \texttt{testAuditor}.

\subsection{\texttt{controller}}
Als \texttt{controller} wird jene zentrale Komponente beschrieben, welche als einzige Komponente direkt mit dem \texttt{user} interagiert, und sich mit der Ausführung der einzelnen Komponenten des Verfahren beschäftigt, sowie I/O Prozesse der Komponenten und Fehlerereignisse verwaltet.\\
Typischerweise erfüllt eine CI-Pipeline die Rolle des \texttt{controllers}, allerdings kann der \texttt{controller} auch auf andere Weisen implementiert werden.

\subsection{\texttt{user}}
Die Bezechnung \texttt{user} ist bewusst breit interpretierbar gehalten. Sie beschreibt eine externe Instanz, die mit dem \texttt{controller} interagiert, so den Ablauf des Verfahrens anstößt und/oder die Ausgabe des Verfahrens konsumiert. Dabei kann es sich um einen menschlichen Benutzer, oder um automatisierte Prozesse handeln. 

\section{Anforderungen an ein Testframework}\label{chap:req_testframework}
Das beschriebene Verfahren kann mit jedem Testframework angewandt werden, welches diese Voraussetzungen erfüllt, die sich unter Anderem auf \chapref{chap:conclusion_testformat_eval} ergeben:
\begin{itemize}
    \item Eine beliebig kleine Untermenge aller Testcases kann gezielt ausgeführt werden
    \item JUnit XML wird als Ausgabeformat für Testdurchläufe verwendet
    \item Ein Testcase kann eindeutig über die Kombination der Attribute \texttt{name} und \texttt{classname} identifiziert werden
\end{itemize}

\section{Anforderungen an ein Projekt}\label{chap:Anforderungen_Framework}
Nachdem die Anforderungen und Funktionsweisen der beteiligten Komponenten und Dateien auführlich diskutiert wurden, können daraus nun die Anforderungen an ein Projekt abgeleitet werden, die vorausgesetzt werden, um das Verfahren implementieren zu können:
\begin{itemize}
    \item Determinismus ist gegeben - Jeder Testdurchlauf erzeugt bei gleichem Input und gleicher Konfiguration der Testcases dasselbe Ergebnis, unabhängig von der Entwicklungsumgebung
    \item Das verwendete Testframework erfüllt alle in \chapref{chap:req_testframework} dargestellten Anforderungen
    \item Es werden Komponenten zur Verfügung gestellt, welche die in \chapref{chap:Komponenten} beschriebenen Funktionalitäten bereitstellen
\end{itemize}
